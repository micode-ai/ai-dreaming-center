# Last-scan indicator on findings & ideas pages

**Date:** 2026-05-31
**Status:** Approved (design)
**Author:** brainstorming session

## Problem

The findings (tech-debt / «долги») and ideas («идеи») pages list items that are
generated by a scan slash-command (`/tech-debt-scan`, `/product-idea-scan`,
triggered by the "Run scan" button at the top of each page). Nothing on the page
tells the user **when the list was last scanned** or whether the last scan
succeeded — so a stale or empty list is indistinguishable from a fresh one.

The user wants each page to **display the last scan run** — its status and time.

> Clarification captured during brainstorming: "последний запуск" means the last
> run of the **scan command** that regenerates the list — NOT the per-item
> `orchestration_run` backlink. The per-item run link in the refs column is
> unrelated and unchanged.

## What the data already provides (verified)

`ProcessManager.start_command(...)` records every scan run as a row in
`agent_learning_sessions`:

- `agent_name = "cmd:{project.slug}:tech-debt-scan"` (findings) /
  `"cmd:{project.slug}:product-idea-scan"` (ideas).
- columns: `started_at`, `finished_at`, `status` (`running` → `success` /
  `failed` on cleanup), `error_message`.

So the last scan = the most-recent `agent_learning_sessions` row for that
project + command name. **No schema change is needed.**

The shared partial `_scan_action_bar.html` exposes the macro
`scan_action_bar(action_url, running, label, running_label, hint='')` and is used
by findings, ideas, wiki, and wiki-health. Both findings and ideas already compute
`scan_running` (whether the command is currently in `pm.list_running()`).

## Design

### Behaviour

Next to the scan button (top of the findings and ideas pages), show a compact
indicator:

- **Status badge** colored by the last run's status: `running` (or live
  `scan_running`), `success`, `failed`. When no scan row exists →
  "не запускался / never scanned" (no badge color).
- **Time**: relative ("3 мин назад") as the visible text, with the exact
  timestamp in the element's `title` (hover). For a finished run use
  `finished_at`; for a still-running one use `started_at`.
- If the last row's status is `failed` and it carries an `error_message`, the
  badge's `title` may include it (optional, nice-to-have — not required).

Relative time is rendered client-side with the browser's built-in
`Intl.RelativeTimeFormat(locale)` over an ISO timestamp embedded in a
`data-*`/`<time datetime>` attribute. Rationale: locale-aware pluralization for
free, zero new dependencies, no server-side TZ/pluralization logic. The exact
timestamp (server-rendered, sliced to minute precision) is the `title` fallback
and is what shows if JS is disabled.

`scan_running` (from `pm.list_running()`) takes precedence over the DB row's
status for the "is it running right now" determination — the DB row may still say
`running` after a crash, and conversely a brand-new run appears in
`pm.list_running()` immediately.

### Components & boundaries

- **`db.py`** — new helper `get_last_command_session(project_id, command_name)`:
  one query, `SELECT status, started_at, finished_at, error_message FROM
  agent_learning_sessions WHERE project_id=? AND agent_name=? ORDER BY started_at
  DESC LIMIT 1`, where `agent_name = f"cmd:{slug}:{command_name}"` (the caller
  passes the already-composed key, OR passes slug+command and the helper composes
  it — plan decides; prefer composing inside the route to keep db.py slug-agnostic).
  Returns a dict or `None`. Single responsibility: last-command lookup.
- **`routes/project_findings.py`** — `findings_page` fetches the last
  `tech-debt-scan` session and passes `last_scan` into the template context.
- **`routes/project_ideas.py`** — `ideas_page` does the same for
  `product-idea-scan`.
- **`_scan_action_bar.html`** — add an OPTIONAL macro parameter
  `last_scan=None`. When provided, render the badge + `<time>` element. When
  omitted (wiki / wiki-health callers), render exactly as today — **backward
  compatible**. The tiny `Intl.RelativeTimeFormat` script lives with this partial
  (guarded so multiple includes don't double-bind).
- **i18n** — `scan.last.*` keys (`never`, `running`, plus status labels if not
  already shared), RU + EN mirrored, via the Write/Edit tool (UTF-8). Verified by
  `scripts/check_i18n.py`.

### Data flow

`route → get_last_command_session → {status, started_at, finished_at} → template
context last_scan → scan_action_bar(..., last_scan=last_scan) → badge + <time
data-ts=finished_at|started_at> → client JS fills relative text`.

### Error handling

- No scan row → indicator shows "never scanned"; no crash.
- `get_last_command_session` wrapped so a DB error degrades to `last_scan=None`
  (indicator simply absent / "never"), never breaks the page.
- Malformed/empty timestamp → JS leaves the server-rendered absolute text as-is.

## Testing

No unit-test suite (by project design). Verification:

- A smoke check (extend an existing `scripts/smoke_*` or a tiny new one) that
  `get_last_command_session` returns the most-recent row for a given
  project+command and `None` when absent.
- `scripts/check_i18n.py` for key parity.
- Manual: run a scan on findings/ideas, confirm the indicator shows `running` then
  `success` with a sensible relative time; confirm wiki / wiki-health pages
  (which don't pass `last_scan`) are visually unchanged.

## Out of scope (YAGNI)

- Full scan history / log of all past scans.
- Re-running the scan from the indicator (the button already does that).
- Progress / ETA / live countdown.
- Touching the per-item `orchestration_run` refs link.
- Adding the indicator to wiki / wiki-health (not requested; the optional param
  leaves them untouched).
