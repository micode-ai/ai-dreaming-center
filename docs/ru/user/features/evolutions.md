# Evolutions — предложения правок для агентов

`/p/{slug}/evolutions` — таблица evolution-предложений: markdown'ов с правками для конкретных агентов (`.claude/agents/<name>.md`). Их пишет slash-команда `/evolve-agent` во время self-study — агент сам решает, что у него в инструкциях стоит улучшить, и кладёт предложение в `evolutions_dir` (по умолчанию `.claude/agents/_context/`).

## Откуда берутся

Evolution-файлы появляются:
- Когда self-study-сессия агента запущена с кнопки **Start session** на `/p/{slug}/rotation` или ночным cron'ом `nightly_learning`.
- Во время сессии `/self-study {agent}` агент использует slash-команду `/evolve-agent`, которая пишет markdown в `evolutions_dir`.
- Отдельной кнопки «сгенерировать эволюции» нет — они побочный продукт self-study.

Имя файла обычно `YYYY-MM-DD-<slug>.md`, frontmatter содержит как минимум `agent`, `status`, опционально `title`, `conflict`, `github_issue`, `orchestration_run`.

## Таблица предложений

Колонки:
- **агент** — имя `.claude/agents/<agent>.md`, к которому относится предложение.
- **имя** — короткое имя файла (без `.md`). Кликабельно — открывает модалку с rendered markdown'ом + ссылку на raw.
- **статус** — select: `proposed` / `active` / `applied` / `archived` / `rejected`. Меняется submit'ом формы (auto-submit on change).
- **конфликт** — `⚠` если есть пересечение с другим открытым предложением на того же агента.
- **заголовок** — `title` из frontmatter.
- **ссылки** — `GH` (ссылка на github_issue) + `run` (ссылка на orchestration_run).
- **(actions)** — кнопки Apply / → GH / del / raw.

## Фильтры

Под заголовками — строка фильтров:

| Колонка | Фильтр |
|---|---|
| агент | поиск по подстроке |
| имя | поиск по подстроке |
| статус | селект (все / 5 статусов) |
| конфликт | селект (все / только конфликт / без конфликта) |
| заголовок | поиск по подстроке |
| ссылки | селект (все / есть GH / есть run / без ссылок) |
| (actions) | кнопка «сбросить» |

Состояние сохраняется в `localStorage` под ключом `evolutions.filters.{slug}` — отдельно для каждого проекта.

## Apply — применить предложение

Кнопка **«Применить»** (фиолетовая) — POST `/p/{slug}/evolutions/apply` с relative_path файла. После клика — модалка подтверждения:

> Запустить Orchestrator: применить эту правку к `.claude/agents/<agent>.md`?

После «Подтвердить» — спавнится Orchestrator-run с goal'ом:
1. Прочитать текущий агент-файл.
2. Применить раздел «Proposed change» evolution-файла.
3. При конфликте с другими разделами — добавить «## Open questions» вместо переписывания.
4. Обновить frontmatter evolution-файла на `status: applied` + `applied_at`.
5. Завершить run.

В frontmatter evolution-файла записывается `orchestration_run: <run_id>` — потом видно по ссылке `run` в колонке «ссылки».

Редирект — на `/p/{slug}/orchestration/{run_id}`, можно наблюдать прогресс.

## Conflict-gate и force-apply

Если ≥2 открытых evolution-предложения нацелены на одного агента — все они автоматически помечаются `has_conflict=true` (даже без явного `conflict: true` во frontmatter). Это защита от того, что одновременное применение «затрёт» друг друга.

В таблице у таких строк:
- В колонке «конфликт» — `⚠`.
- Кнопка Apply становится **красной** и подписана «Применить (force)».
- В hidden-поле формы `force=1`.
- Confirm-модалка превращается в danger:

  > ⚠ КОНФЛИКТ: ещё одно открытое evolution-предложение нацелено на агента «<agent>». Применить всё равно? Это может перетереть параллельные изменения.

  Кнопка OK называется «Применить (force)», не «Удалить» (хоть variant и danger).

Сервер дублирует gate: POST `/evolutions/apply` без `force=1` на конфликтном файле возвращает HTTP 409 с понятным сообщением — даже если фронтенд решит прислать запрос напрямую.

### Что делать с конфликтами

- **Решить вручную**: пройтись по файлам, выбрать один, остальные пометить `status: rejected` или удалить. Тогда `has_conflict` спадает автоматически.
- **Применить force-apply** — если ты уверен, что они дополняют друг друга или старые предложения уже неактуальны.
- **Поставить `conflict: false` во frontmatter** — отметка «я проверил, конфликта нет на самом деле». После этого Apply работает без force.

## Создать GitHub issue

Кнопка **«→ GH»** — POST `/p/{slug}/evolutions/github`. Создаёт GitHub issue из контента evolution-файла (через `gh` CLI). После успеха в frontmatter появляется `github_issue: <url>`, кнопка пропадает, появляется ссылка `GH` в колонке «ссылки».

Если `gh` не настроен или нет доступа к репо — операция упадёт с понятной ошибкой.

## Сменить статус

Селект «статус» (auto-submit) — POST `/p/{slug}/evolutions/status` переписывает `status:` в frontmatter. Используй чтобы:
- Архивировать неактуальное (`archived`).
- Отвергнуть откровенно плохие (`rejected`).
- Пометить как уже применённое вручную (`applied`).

## Удалить

Кнопка **«del»** (красная) — POST `/p/{slug}/evolutions/delete`. Confirm-модалка:

> Удалить файл `<relative_path>`?

Кнопка OK — «Удалить» (variant danger). Файл удаляется с диска (unlink) — git-историю это не трогает, но если он не закоммичен — пропадёт навсегда.

## Массовый запуск

Слева у каждой строки — чекбокс. Чекбокс в заголовке — выбрать все **видимые** строки (отфильтрованные не трогаются). Кнопка **«Запустить (N)»** под счётчиком ставит выбранное в очередь Orchestrator'а — там оно идёт последовательно.

Особенности bulk для эволюций:
- Если в выборку попали конфликтные — они упадут с `failed` (conflict без force).
- На странице `/p/{slug}/orchestration` в баннере очереди есть кнопка **«повторить с force (N)»** — флипает failed обратно в pending с `force=1`. Это то, что нужно, если вы намеренно хотите применить пачку конфликтующих эволюций.

Подробнее про очередь, диспатчер, retry — [orchestration.md → Bulk queue](orchestration.md#bulk-queue--массовый-запуск).

## Rubric-блок

Сверху страницы (если есть данные) — панель **«Rubric агрегация»**: распределение verdict'ов из rubric-блоков evolution-файлов. Колонки:
- **auto** (зелёный) — `verdict: auto-apply` (можно применять автоматически).
- **review** (амбер) — `verdict: review` (нужно ручное ревью).
- **reject** (красный) — `verdict: reject` (отклонить).
- **incomplete** (серый) — rubric-блок есть, но не заполнен.

Если ни в одном файле нет rubric-блока — выводится подсказка: «Ни в одном файле нет rubric-блока. /evolve-agent старых версий не пишет rubric — это ожидаемо. Новые предложения с frontmatter rubric: попадут сюда.»

Это **информационная** панель — на действия не влияет, просто показывает «здоровье» предложений.
