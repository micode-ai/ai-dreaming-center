## Цель

AI Dreaming Center (DC) — это единая панель управления командой Claude-агентов на нескольких проектах сразу. Если у тебя есть пять-десять локальных репозиториев, и в каждом ты держишь свою команду агентов через `.claude/agents/` — DC даёт тебе один UI поверх всего этого.

Что ты получаешь через DC:

- Запускать ночные self-study прогоны (агент читает свои инструкции, кодовую базу, и пишет конспект).
- Видеть live-стрим логов работающей сессии.
- Читать конспекты, которые агенты писали по ночам.
- Смотреть tech-debt и продуктовые идеи, которые агенты накидали (и одной кнопкой создавать из идеи Jira-тикет).
- Запускать оркестратора Романа на свободно-формулированную цель и смотреть, как он декомпозирует её на под-агентов.
- Смотреть аналитику — сколько токенов и денег ушло на каждый проект.

DC заменяет ад из непрозрачных n8n-воркфлоу и `cron`-скриптов одной FastAPI-страничкой.

## Концептуальная модель

Главные сущности:

- **Проект** (project) — одна папка под `projects_root`. Например, если `projects_root = D:\Work\micode\`, то `D:\Work\micode\my-app\` — один проект, `D:\Work\micode\wishlist\` — второй. У каждого проекта свой `.claude/agents/` каталог с агентами и своя статистика обучения в БД DC.
- **Агент** (agent) — один markdown-файл в `.claude/agents/{agent-name}.md` внутри проекта. Описывает поведение Claude.
- **Сессия** (session) — один запуск Claude CLI в режиме self-study для одного агента. У сессии есть статус (success / failed / timeout), длительность и опциональный путь к конспекту.
- **Run** (orchestration run) — оркестрация Романа: одна цель, дерево из root-агента и под-агентов.
- **Cascade** — структурированный run с фазами contract → design → implementation → review → qa и gate-verdict'ами между ними.

Все сущности изолированы между проектами. Удаление проекта каскадно сносит всю его историю.

## Навигация

- Корень `/` — aggregated dashboard поверх всех включённых проектов: top-line метрики (success/failed/timeout за неделю, running прямо сейчас, total tech-debt, total ideas) + карточки проектов.
- `/projects` — registry проектов: таблица со списком, кнопки `Disable` / `Delete`, форма «Просканировать и импортировать новые».
- `/p/{slug}/` — dashboard конкретного проекта: его метрики, последние сессии, активные runs.
- `/p/{slug}/{раздел}` — конкретные разделы (orchestration, live, rotation, topics, kanban, notes, findings, tech-debt, ideas, wiki, ai-usage, evolutions, loops, plans, cascade-costs, contracts, sidecar-findings, settings).
- `/settings` — глобальные настройки (~92 ключа в 13 группах).
- `/p/{slug}/settings` — per-project overrides поверх глобальных.

В шапке всегда есть `<select>` со списком всех проектов: переключаешься быстро. Кнопка `EN/RU` справа меняет язык интерфейса.

## Какие сценарии решает

- **Nightly self-study**: каждую ночь cron берёт из rotation top-N агентов с самым старым `last_studied_at` и запускает их по очереди. Утром у тебя на dashboard свежий список конспектов.
- **Ручной прогон агента**: тебе срочно нужно, чтобы какой-то агент перечитал свои инструкции — открываешь rotation проекта, нажимаешь `Start session` рядом с ним, ждёшь.
- **Просмотр tech-debt**: weekly scanner написал десяток markdown-файлов в `tech_debt_dir`; ты открываешь `findings`, видишь список, кликаешь по интересному, прокручиваешь — close или delete.
- **Идея → Jira**: видишь хорошую идею в product_ideas, нажимаешь `→ Jira`, тикет создаётся в твоём Jira-проекте.
- **Оркестрация цели**: «декомпозируй фичу X для модуля Y» — пишешь это в форму на `/p/{slug}/orchestration`, Roman поднимает run, ты на detail-странице смотришь, как появляются под-агенты и их сообщения.
- **Аналитика затрат**: открываешь `/ai-usage` (global) или `/p/{slug}/ai-usage` — видишь tokens/cost по моделям и проектам.

## Что DC не умеет

Чтобы не было иллюзий — DC это **control panel**, а не редактор кода и не CI-сервер:

- DC **не редактирует** твой код. Code-mod'ы делает Claude в своих сессиях.
- DC **не запускает тесты**. Это делает Claude (если попросишь) или твой CI.
- DC **не управляет deploy**'ем. Никаких кнопок «Deploy to prod».
- DC **не агрегирует git-историю и не делает PR**. Это Claude через свои tool-use'ы.
- DC **не хранит код агентов**. Они живут в `.claude/agents/` твоих проектов; DC только дёргает их через slash-команды.
- DC **не редактирует** twoи markdown-артефакты (tech-debt, ideas) сам — он только их читает и переписывает frontmatter (`status: closed`, `jira_ticket: ABC-123`). Содержимое body неизменно.

## Ограничения для пользователя

- **Один процесс claude на агент-проект**: попытка стартовать вторую сессию для того же агента в том же проекте — отклонится с «уже запущен».
- **Глобальный лимит параллелизма**: `max_concurrent` (default 1) — даже если ты в разных проектах нажмёшь `Start session` сразу для трёх агентов, они выстроятся в очередь.
- **Per-project лимит**: опционально, override через настройки.
- **Watchdog**: каждая сессия ограничена `timeout_minutes` (default 20). Долгие задачи через self-study не сделать — это для коротких review-сессий.
- **One Roman per project**: попытка стартовать второй orchestration-run при running-первом — редиректит на existing.

Если тебе хочется глубже понять, как DC устроен под капотом — открой [`../architecture.md`](../architecture.md).
